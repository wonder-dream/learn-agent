## 具体任务

### 1. 博客结构 (0.5h 规划)

```
标题：从零构建一个代码审查 Agent：AST + LLM 混合架构实践

1. 为什么做代码审查 Agent？（200字）
   - 现有工具的局限（只做静态、不懂语义）
   - 纯 LLM 的问题（幻觉、成本高、慢）
   - 混合架构的思路

2. 架构设计（500字 + 架构图）
   - 静态分析层（11 条规则）
   - LLM 精判层（4 种检测类型）
   - LangGraph 编排
   - 技术选型理由

3. 关键实现细节（800字）
   - AST 静态分析：规则引擎设计
   - 候选人提取：不喂全文件，只喂可疑片段
   - LLM 精判：Prompt 设计 + 温度参数
   - 错误处理：重试 + 超时 + 降级

4. 效果数据（300字）
   - 测试方法（10 个开源文件）
   - 量化结果（准确率 87%、单文件 ¥0.03）
   - 误报/漏报分析

5. 总结与后续（200字）
   - 当前局限
   - 后续方向（多 Agent 分工、GitHub App 集成）

6. 附录：GitHub 仓库链接
```

### 2. 写作 (2.5h)

写作要点：
- 不要写"调 API"的部分——每个人都会，写了掉价
- 重点写**架构决策和 tradeoff**（为什么用 LangGraph 不用 LangChain？为什么先静态再 LLM？）
- 每个技术决策后面跟一个"为什么"
- 量化数据必须出现——这是文章最有说服力的部分
