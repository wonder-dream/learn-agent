### 1. System Prompt 黄金结构 (1h)

一个好的 System Prompt 包含四要素：

```markdown
## 角色 (Role)
你是世界顶级的 Python 代码审查专家，拥有 10 年经验。

## 任务 (Task)
审查给定的 Python 代码，发现潜在问题并按严重程度分类报告。

## 输出格式 (Output Format)
你必须以 JSON 格式输出，结构如下：
```json
{
    "issues": [
        {"line": int, "severity": "error|warning|info", "category": "style|naming|logic|security", "message": str}
    ],
    "summary": str
}
```

## 约束 (Constraints)
- 不要报告主观风格偏好（如"缩进用 tab 还是空格"）
- 每个 issue 的 message 必须一句话说清楚问题 + 建议
- 如果没有发现问题，返回空 issues 列表
```

四要素缺一不可：
- **角色**：定调，影响模型输出风格
- **任务**：明确要做什么
- **格式**：结构化输出是工程化的基础（可解析、可统计）
- **约束**：减少幻觉、控制输出范围

### 2. CoT 与变体 (1h)

**零样本 CoT**：直接问问题时加上"我们一步步思考"：
```python
prompt = """
请审查下列代码：
{code}

在给出最终结论之前，请：
1. 先列出所有变量和函数
2. 逐一检查命名规范
3. 逐一分析是否有空指针/越界风险
4. 汇总以上发现，生成最终 Review
"""
```

**Few-shot**：给 1-3 个示例，模型照猫画虎：
```python
examples = """
Example 1:
Input:
def add(a,b): return a+b
Output:
{"issues": [{"line":1, "severity":"warning", "category":"style", "message":"PEP 8: missing spaces after comma"}]}

Example 2:
Input:
def divide(x,y): return x/y if y!=0 else None
Output:
{"issues": [
    {"line":1, "severity":"warning", "category":"style", "message":"PEP 8: missing spaces after comma"},
    {"line":1, "severity":"info", "category":"logic", "message":"Division by zero handled, but returning None may cause NoneType errors downstream"}
]}
"""
```

### 3. 写代码审查 Agent 的完整 Prompt (1.5h)

文件组织：
```
projects/code-review-agent/src/prompts/
├── __init__.py
├── system_prompt.py    # 审查 Agent 的 System Prompt
└── templates.py        # Prompt 模板（支持变量注入）
```

```python
# src/prompts/system_prompt.py

CODE_REVIEW_SYSTEM = """你是世界顶级的 Python 代码审查专家。

## 职责
逐行审查给定的 Python 代码，发现并报告所有潜在问题。

## 审查维度
1. **命名规范**：变量/函数/类名是否符合 PEP 8
2. **代码风格**：缩进、空格、行长度、注释
3. **逻辑错误**：空指针风险、边界条件遗漏、类型错误
4. **安全隐患**：SQL 注入、路径遍历、不安全反序列化

## 输出格式
```json
{
    "issues": [
        {
            "line": <int>,
            "severity": "error" | "warning" | "info",
            "category": "style" | "naming" | "logic" | "security",
            "message": "<问题描述 + 修复建议>",
            "suggestion": "<修复后的代码片段>"
        }
    ],
    "summary": "<一句话总结>",
    "total_issues": <int>
}
```

## 约束
- 每个 issue 必须包含具体行号
- 风格问题 severity 最高为 "warning"
- 安全漏洞 severity 必须是 "error"
- 不确定的问题 severity 设为 "info" 并注明不确定性
- 严禁无中生有（看到问题才报，不要猜测）
"""
```
