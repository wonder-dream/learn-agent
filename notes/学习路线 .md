## 第一部分：基础重塑 —— asyncio 不是语法糖

在深入 Agent 或 AI Infra 之前，你必须对 asyncio 有**系统级**理解。大部分人的 asyncio 知识停留在"用 `async`/`await` 写非阻塞 HTTP 请求"，这远远不够。

### 1.1 事件循环（Event Loop）的源码级理解

Python 的 `asyncio` 基于**事件循环 + 协程调度队列**。核心组件：

- **Selector**：底层使用 `epoll`（Linux）、`kqueue`（macOS）或 `IOCP`（Windows）做 IO 多路复用。
- **Task**：包装协程，维护状态（pending/running/finished），注册回调。
- **Handle**：对回调函数的包装，包含回调、参数、上下文。
- **Future**：表示异步操作的最终结果，Task 是 Future 的子类。

**关键源码路径**（CPython 源码）：
```
Modules/selectmodule.c      # epoll/kqueue 封装
Lib/asyncio/base_events.py  # BaseEventLoop
Lib/asyncio/tasks.py        # Task 实现
```

**必须理解的问题**：
- `loop.run_until_complete()` 怎么把协程注册到事件循环？
- `Task._step()` 怎么通过 `send(None)` 驱动协程执行到下一个 `await`？
- 当协程遇到 `await asyncio.sleep(0)` 时，控制权怎么交还给事件循环？

### 1.2 协程调度机制

事件循环的一次迭代（`run_once`）大致流程：
1. 执行**就绪队列**（`_ready`）中的所有回调（Handle）；
2. 通过 Selector 获取**已就绪的 IO 事件**（文件描述符可读/可写）；
3. 将对应的回调加入就绪队列；
4. 处理**延迟调用**（`call_later` 注册的定时器）。

**面试常问**：
- 为什么 `asyncio` 是**单线程**的？多核 CPU 怎么利用？（答：IO 密集型用 `asyncio`，CPU 密集型用 `ProcessPoolExecutor` 或 `run_in_executor`）
- `asyncio` 的协程切换是**协作式**还是**抢占式**？（答：协作式，只有遇到 `await` 才会让出控制权）

### 1.3 同步原语深度解析

| 原语                | 用途     | 生产级注意点                                         |
| ------------------- | -------- | ---------------------------------------------------- |
| `asyncio.Lock`      | 互斥锁   | 必须注意**锁的公平性**，默认非公平，高并发下可能饿死 |
| `asyncio.Semaphore` | 限流     | 生产环境需要**动态调整 limit**，而不是写死           |
| `asyncio.Queue`     | 消息队列 | `maxsize` 是背压的核心，满时 `put()` 阻塞或抛异常    |
| `asyncio.Event`     | 事件通知 | 注意 `clear()` 的时序，防止信号丢失                  |
| `asyncio.Condition` | 条件变量 | 必须配合 `while` 循环使用，防止虚假唤醒              |

**关键陷阱**：`asyncio.Semaphore` 的默认实现是**非公平**的。在 LLM API 限流场景下，如果 100 个协程同时 `acquire`，后到的可能先获得锁，导致先到请求被饿死。生产级实现需要自研**公平队列**或**优先级队列**。

### 1.4 性能陷阱与调试

- **大量 Task 创建**：`asyncio.create_task()` 会立即调度，如果在一个循环内创建 10 万个 Task，会导致事件循环阻塞。应使用**任务池**或**批量提交**。
- **内存泄漏**：Task 的异常如果没有被 `await` 或 `add_done_callback` 处理，Python 会保留异常 traceback，导致内存泄漏。务必在每个 Task 上调用 `add_done_callback(lambda t: t.exception())`。
- **CPU 密集型代码阻塞事件循环**：`json.loads` 处理大 JSON、`numpy` 计算等都会阻塞。应使用 `loop.run_in_executor()` 或 `asyncio.to_thread()`（Python 3.9+）。

## 第二部分：Agent 开发方向

### 2.1 核心架构认知：从 LangChain 到 LangGraph

**LangChain 的问题**：Chain 是**线性**的，不适合多 Agent 协作。Agent 的 Tool Use 是**阻塞式**的，无法并行调用多个 Tool。

**LangGraph 的革新**：引入 **Pregel 图执行模型**，支持：
- **状态机**：节点共享一个全局状态，通过 `Channels` 管理状态合并；
- **并行执行**：同一 superstep 内的多个节点可以并行执行；
- **条件边**：根据状态决定下一个执行节点；
- **持久化**：支持 checkpoint，支持断点续跑。

#### 2.1.1 Pregel 执行模型源码解析

LangGraph 的 `Pregel` 类（`langgraph/pregel/__init__.py`）是核心执行引擎。它的执行流程：

1. **编译阶段**：`StateGraph.compile()` 将用户定义的图编译成 `CompiledGraph`，生成 `channels`（状态通道）和 `nodes`（执行节点）。
2. **初始化阶段**：`Pregel.astream()` 创建初始状态，将所有节点标记为**就绪**。
3. **Superstep 循环**：
   - **Step 1 - 调度**：找出所有**入边已满足**的节点（即所有依赖节点的状态已更新）。
   - **Step 2 - 并行执行**：使用 `asyncio.gather()` 并行执行所有就绪节点。每个节点是一个异步函数，接收当前状态快照，返回状态增量。
   - **Step 3 - 状态合并**：通过 `Channels` 的 `update` 方法合并所有节点的状态增量。`Topic` 通道会追加，`LastValue` 通道会覆盖。
   - **Step 4 - 条件判断**：检查是否有条件边（`ConditionalEdge`），根据状态决定下一个 superstep 的入口节点。
   - 循环直到没有就绪节点。

**关键源码阅读清单**：
```python
# langgraph/pregel/__init__.py
class Pregel:
    async def astream(self, input, config=None, ...):
        # 1. 初始化状态
        # 2. 进入 superstep 循环
        while True:
            # 3. 找出就绪节点
            tasks = [self._arun_node(node, state) for node in ready_nodes]
            # 4. 并行执行
            results = await asyncio.gather(*tasks, return_exceptions=True)
            # 5. 合并状态增量
            for result in results:
                for channel, value in result.items():
                    channels[channel].update(value)
            # 6. 判断终止条件
            if not ready_nodes:
                break
```

**面试必问**：
- **Q**：如果两个节点在同一个 superstep 内修改了同一个 `LastValue` 通道，结果是什么？  
  **A**：`LastValue` 通道在 superstep 结束时只保留**最后一个**更新（按节点执行顺序），这会导致竞态。应使用 `Topic` 通道或自定义 `reducer` 函数合并增量。

- **Q**：`fan-out/fan-in` 在 LangGraph 中怎么实现？  
  **A**：`fan-out` 通过从一个节点引出多条边实现；`fan-in` 通过多个节点指向同一个下游节点实现。下游节点会在所有上游节点都执行完后（即所有入边状态都更新后）才被调度。

#### 2.1.2 状态管理（Channels）深度理解

LangGraph 的 `Channels` 是状态管理的核心抽象：

| Channel 类型              | 行为               | 适用场景                   |
| ------------------------- | ------------------ | -------------------------- |
| `LastValue`               | 覆盖旧值           | 单值状态（如当前用户输入） |
| `Topic`                   | 追加新值           | 消息历史（如对话记录）     |
| `EphemeralValue`          | 只读一次，读后清空 | 临时事件（如中断信号）     |
| `BinaryOperatorAggregate` | 通过二元操作符聚合 | 计数器、累加器             |

**源码位置**：`langgraph/channels/`

**关键问题**：`EphemeralValue` 的"读后清空"是在什么时候发生的？（答：在下一个 superstep 的**读取阶段**，通过 `consume()` 方法清空，确保事件只被处理一次。）

#### 2.1.3 流式架构设计

LangGraph 的流式输出不是简单的 `print`，而是通过 `StreamWriter` 发送结构化事件：

```python
# 节点内部调用
from langgraph.types import StreamWriter
writer = StreamWriter()
writer({"type": "tool_start", "tool": "search", "query": "..."})
```

**流式模式**：
- `stream_mode="updates"`：每个 superstep 结束后输出状态增量；
- `stream_mode="custom"`：输出节点内部通过 `StreamWriter` 发送的自定义事件；
- `stream_mode="messages"`：输出 LLM 的 token 流。

**生产级注意**：`StreamWriter` 的事件会缓存在内存中，如果前端消费慢，会导致内存堆积。需要实现**背压机制**（如 `asyncio.Queue(maxsize=100)`，满时丢弃旧事件或阻塞写入）。

### 2.2 多 Agent 系统：消息总线与协调器

生产级多 Agent 系统不是简单的 `asyncio.gather([agent1(), agent2()])`，而是需要**消息总线解耦**。

#### 2.2.1 消息总线设计

核心组件：
- **Message Schema**：`AgentMessage(sender, receiver, msg_type, content, task_id, timestamp, priority)`
- **Message Bus**：基于 `asyncio.Queue` 的内存总线（单机）或 Redis Streams（分布式）
- **Router**：根据 `receiver` 字段路由消息，支持点对点、广播（`receiver="*"`*）、发布订阅（`topic` 匹配）

**背压实现**：
```python
class BackpressureMessageBus:
    def __init__(self, maxsize=1000):
        self.queue = asyncio.Queue(maxsize=maxsize)
        self.dropped_count = 0
    
    async def publish(self, msg: AgentMessage):
        try:
            self.queue.put_nowait(msg)
        except asyncio.QueueFull:
            # 生产级策略：丢弃低优先级消息
            if msg.priority == Priority.LOW:
                self.dropped_count += 1
                return
            # 或阻塞等待
            await self.queue.put(msg)
```

#### 2.2.2 分层协调器（Orchestrator）

生产级多 Agent 系统通常采用**三层架构**：

1. **管理 Agent（Manager Agent）**：负责任务分解，将用户请求拆分为子任务 DAG；
2. **协调层（Coordinator）**：负责任务调度（拓扑排序、依赖管理、错误重试、超时熔断）；
3. **执行层（Worker Agent）**：负责具体执行（Tool Use、LLM 调用、代码执行）。

**协调器核心逻辑**：
```python
class Coordinator:
    async def execute_dag(self, dag: DAG):
        # 拓扑排序
        ready_tasks = dag.get_ready_tasks()
        while ready_tasks:
            # 并行执行所有就绪任务
            tasks = [self._execute_task(t) for t in ready_tasks]
            results = await asyncio.gather(*tasks, return_exceptions=True)
            
            # 处理结果
            for task, result in zip(ready_tasks, results):
                if isinstance(result, Exception):
                    if task.retry_count < 2:
                        task.retry_count += 1
                        dag.requeue(task)
                    else:
                        dag.mark_failed(task)
                else:
                    dag.mark_done(task, result)
            
            ready_tasks = dag.get_ready_tasks()
```

**超时熔断**：
```python
async def _execute_task(self, task: Task):
    try:
        return await asyncio.wait_for(
            self.worker.execute(task),
            timeout=task.timeout
        )
    except asyncio.TimeoutError:
        # 熔断：标记该 worker 为不可用，触发降级
        self.circuit_breaker.record_failure(task.worker_id)
        return FallbackResult()
```

#### 2.2.3 通信模式对比

| 模式     | 实现                                  | 优点                   | 缺点                   | 适用场景                |
| -------- | ------------------------------------- | ---------------------- | ---------------------- | ----------------------- |
| 直接对话 | Agent A 直接 `await agent_b.run(msg)` | 简单、延迟低           | 紧耦合、无法扩展       | 2-3 个 Agent 的简单协作 |
| 消息总线 | 通过 Queue/Redis 中转                 | 解耦、可扩展、可持久化 | 延迟稍高、需要序列化   | 10+ Agent 的系统        |
| 发布订阅 | Agent 订阅 topic，广播接收            | 灵活、支持动态加入     | 消息可能丢失、需要 ACK | 事件驱动架构            |

### 2.3 MCP 协议深层实现

MCP（Model Context Protocol）是 2025-2026 年 Agent 开发的**事实标准**。面试不会问你"怎么用 MCP client"，而是问"怎么实现一个生产级 MCP server"。

#### 2.3.1 协议层解析

MCP 基于 **JSON-RPC 2.0**，核心消息类型：
- **Request**：`{jsonrpc: "2.0", id: string|number, method: string, params: object}`
- **Notification**：无 `id` 的请求（不需要响应）
- **Response**：`{jsonrpc: "2.0", id: ..., result: ...}` 或 `error`

**生命周期**：
1. **初始化**：Client 发送 `initialize`，Server 返回 capabilities（支持的 tools、resources、prompts）；
2. **能力协商**：Client 根据 Server 的 capabilities 决定使用哪些功能；
3. **正常运行**：Tool 调用、Resource 读取、Prompt 获取；
4. **关闭**：Client 发送 `notifications/initialized`，双方关闭连接。

#### 2.3.2 传输层：stdio vs SSE

| 维度       | stdio                          | SSE (Server-Sent Events)         |
| ---------- | ------------------------------ | -------------------------------- |
| 连接方式   | 标准输入输出（子进程）         | HTTP 长连接                      |
| 并发能力   | 低（单进程单线程）             | 高（多连接、多 worker）          |
| 隔离性     | 高（进程隔离）                 | 中（共享进程空间）               |
| 部署复杂度 | 低（本地子进程）               | 高（需要 HTTP 服务）             |
| 适用场景   | 本地工具（如文件系统、数据库） | 远程服务（如搜索 API、计算集群） |

**生产级选择**：
- 对于**计算密集型**或**有状态**的工具（如本地代码执行器），用 `stdio`（进程隔离，崩溃不影响主服务）；
- 对于**高并发**、**无状态**的工具（如搜索、天气查询），用 `SSE`（支持连接池、负载均衡）。

#### 2.3.3 工具注册与并发调用

MCP Server 的工具注册是**动态 schema**：
```python
class MCPServer:
    def __init__(self):
        self.tools: Dict[str, Tool] = {}
    
    def register_tool(self, tool: Tool):
        # 自动生成 JSON Schema
        self.tools[tool.name] = {
            "name": tool.name,
            "description": tool.description,
            "inputSchema": tool.get_json_schema()
        }
    
    async def handle_call_tool(self, request: Request):
        tool = self.tools.get(request.params["name"])
        # 并发控制：每个工具有自己的 Semaphore
        async with tool.semaphore:
            result = await tool.run(request.params["arguments"])
        return result
```

**并发控制**：每个 Tool 实例应持有独立的 `asyncio.Semaphore`，防止单个工具被过度调用（如搜索 API 的 QPS 限制）。

#### 2.4.2 断路器与重试

使用 `tenacity` 库或自研断路器：

```python
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type

class CircuitBreaker:
    def __init__(self, failure_threshold=5, recovery_timeout=30):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.failures = 0
        self.last_failure_time = None
        self.state = "CLOSED"  # CLOSED, OPEN, HALF_OPEN
    
    async def call(self, func, *args, **kwargs):
        if self.state == "OPEN":
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = "HALF_OPEN"
            else:
                raise CircuitBreakerOpen()
        
        try:
            result = await func(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise

# 使用
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=4, max=10),
    retry=retry_if_exception_type((TimeoutError, RateLimitError))
)
async def call_llm_api(prompt: str):
    # ...
```

### 2.5 生产级项目：异步多 Agent 调度引擎

**项目名称**：`AgentFlow` —— 异步多 Agent 调度与执行引擎

**项目架构**：
```
agentflow/
├── core/
│   ├── bus.py          # 异步消息总线（Queue + pub/sub）
│   ├── coordinator.py  # DAG 任务调度器
│   ├── worker.py       # Agent 执行器
│   └── state.py        # 全局状态管理
├── mcp/
│   ├── server.py       # MCP Server 实现
│   └── transport.py    # stdio / SSE 传输层
├── api/
│   ├── main.py         # FastAPI 入口
│   └── stream.py       # SSE 流式输出
└── infra/
    ├── circuit.py      # 断路器
    ├── limiter.py      # 限流器（Token Bucket）
    └── pool.py         # 连接池
```

**核心亮点**：
1. **自研消息总线**：基于 `asyncio.Queue` 实现，支持点对点、广播、发布订阅三种模式，内置背压（`maxsize=1000`，满时丢弃低优先级消息）。
2. **DAG 调度器**：管理 Agent 负责任务分解（调用 LLM 生成子任务 DAG），协调器做拓扑排序和并行调度，执行层 Agent 通过 `asyncio.gather` 并发执行。

## 第三部分：AI Infra 方向

### 3.1 vLLM 源码级解析

vLLM 是 2026 年 AI Infra 面试的**必考项目**。你必须能从头到尾讲清楚一个请求从 HTTP 入口到 GPU kernel 的完整路径。

#### 3.1.1 LLMEngine 入口

`LLMEngine`（`vllm/engine/llm_engine.py`）是用户直接交互的类：

```python
class LLMEngine:
    def __init__(self, ...):
        self.scheduler = Scheduler(scheduler_config, cache_config)
        self.model_executor = GPUExecutor(model_config, ...)
    
    def add_request(self, request_id, prompt, sampling_params):
        # 1. 将 prompt tokenize
        # 2. 创建 Sequence 和 SequenceGroup
        # 3. 将 SequenceGroup 加入 Scheduler 的 waiting 队列
        seq_group = self._create_sequence_group(request_id, prompt, sampling_params)
        self.scheduler.add_seq_group(seq_group)
    
    def step(self):
        # 1. 调度：决定哪些 SequenceGroup 进入当前 batch
        # 2. 执行：调用 ModelExecutor 做推理
        # 3. 后处理：采样生成下一个 token，更新 Sequence 状态
        seq_group_metadata_list, scheduler_outputs = self.scheduler.schedule()
        output = self.model_executor.execute_model(seq_group_metadata_list)
        self._process_model_outputs(output, scheduler_outputs)
```

#### 3.1.2 Scheduler 调度逻辑

`Scheduler`（`vllm/core/scheduler.py`）是 vLLM 的**心脏**。

**核心数据结构**：
- `waiting`：新到达的请求队列；
- `running`：当前正在推理的请求队列；
- `swapped`：被 swap out 到 CPU 的请求队列。

**`schedule()` 方法的核心逻辑**：
1. **处理 running 队列**：检查每个 running 的 `SequenceGroup` 是否完成（生成了 `eos` 或达到最大长度）。完成的移出，未完成的保留。
2. **处理 swapped 队列**：尝试将 swapped 的请求 swap in 到 GPU。如果显存不够，保留在 swapped。
3. **处理 waiting 队列**：尝试将 waiting 的请求加入 running。如果显存不够，尝试 **preempt**（抢占）running 队列中优先级低的请求（swap out 到 CPU）。
4. **构造 batch**：将最终 running 队列中的请求打包成一个 batch，生成 `seq_group_metadata_list`。

**调度决策**：
- **FCFS（First Come First Serve）**：默认策略，保证公平性；
- **Priority**：支持优先级调度（可配置）；
- **Preemption**：当新请求到来但显存不足时，抢占（swap out）部分 running 请求。

**面试必问**：
- **Q**：`swap_out` 和 `swap_in` 的触发条件？  
  **A**：`swap_out` 在**显存不足且需要调度新请求**时触发，将 running 队列中后到的请求（或低优先级）的 KV Cache 写入 CPU 内存；`swap_in` 在**显存空闲**时触发，将 swapped 队列中的请求读回 GPU。

- **Q**：抢占策略是什么？  
  **A**：默认是**交换**（swap），即将被抢占请求的 KV Cache 写入 CPU；另一种策略是**重算**（recompute），即丢弃 KV Cache，重新做 prefill，但会浪费计算。

#### 3.1.3 BlockManager 内存管理

`BlockManager`（`vllm/core/block_manager_v1.py`）管理 **KV Cache 的物理 block**。

**核心概念**：
- **逻辑 block**：从 0 开始的连续编号，每个 block 存储 `block_size` 个 token 的 KV；
- **物理 block**：GPU 显存中的实际 block，通过 block table 映射；
- **Block table**：每个 `Sequence` 维护一个逻辑 block → 物理 block 的映射表。

**分配策略**：
- **首次分配**：`allocate()` 为 prefill 阶段分配初始 block；
- **追加分配**：`append_slot()` 在 decode 阶段追加 token，如果当前 block 满，分配新 block；
- **引用计数**：物理 block 被多个 `Sequence` 共享时（如 beam search），引用计数 > 1，写时复制（Copy-on-Write）。

**内存碎片问题**：
- **内部碎片**：block 内未使用的 slot（如最后一个 block 只存了 2 个 token，剩余 14 个 slot 浪费）；
- **外部碎片**：物理 block 不连续，导致无法分配大连续块（vLLM 通过分页机制避免了外部碎片）。

#### 3.1.4 ModelRunner 执行模型

`ModelRunner`（`vllm/worker/model_runner.py`）负责将调度器的输出转换为 GPU 可以执行的 tensor。

**输入准备**：
1. **Input IDs**：将所有请求的 token 拼接成一个大 tensor（`input_ids`）；
2. **Position IDs**：每个 token 的位置编码；
3. **Attention Mask**：通过 `block_tables` 和 `context_lens` 构造 PagedAttention 需要的输入；
4. **Slot Mapping**：每个 token 对应的物理 block 和 slot 索引。

**执行**：
```python
def execute_model(self, seq_group_metadata_list):
    # 1. 准备输入
    input_tokens, input_positions, attn_metadata = self._prepare_input(seq_group_metadata_list)
    # 2. 调用模型
    hidden_states = self.model(input_tokens, input_positions, attn_metadata)
    # 3. 采样
    output = self.model.sample(hidden_states, sampling_metadata)
    return output
```

#### 3.1.5 PagedAttention 算子

PagedAttention（`vllm/attention/ops/paged_attn.py`）是 vLLM 的核心创新。

**标准 Attention 的问题**：
- 每个请求的 KV Cache 是**连续存储**的，导致：
  - 内存浪费（预分配最大长度，实际使用少）；
  - 无法共享（beam search 时复制大量 KV）；
  - 外部碎片。

**PagedAttention 的解决**：
- 将 KV Cache 分成固定大小的 **block**（如 16 tokens）；
- 通过 **block table** 做逻辑 → 物理映射；
- 注意力计算时，通过 block table 按需读取物理 block。

**核心算子**：
- `paged_attention_v1`：基础实现，每个 thread 处理一个 output token；
- `paged_attention_v2`：优化版，使用 shared memory 缓存 block，减少显存访问。

**面试必问**：
- **Q**：PagedAttention 的 block table 怎么维护？  
  **A**：每个 `Sequence` 有一个 `block_table: List[int]`，存储逻辑 block 对应的物理 block ID。物理 block 由 `BlockManager` 的 `self.block_tables` 维护，支持引用计数和 Copy-on-Write。

- **Q**：为什么 block size 通常设为 16？  
  **A**：权衡**内部碎片**和**内存管理开销**。block 太小，block table 太大，管理开销高；block 太大，内部碎片多。16 是经验值。

### 3.3 分布式推理

#### 3.3.1 Tensor Parallel (TP)

将模型的每一层按**列或行**切分到多个 GPU。

- **列切分（Column Parallel）**：`Y = X @ [W1; W2]`，在 GPU0 算 `X@W1`，GPU1 算 `X@W2`，最后 `all_gather` 拼接；
- **行切分（Row Parallel）**：`Y = [X1, X2] @ [W1; W2]`，在 GPU0 算 `X1@W1`，GPU1 算 `X2@W2`，最后 `all_reduce` 相加。

**通信瓶颈**：`all_reduce` 的通信量。对于 TP=2，每次前向传播需要 2 次 `all_reduce`（列切分后 gather，行切分后 reduce）。

**面试计算题**：
- Q：70B 模型，TP=4，每次 forward 的 `all_reduce` 通信量是多少？  
  A：假设 hidden_size=8192，batch_size=1，seq_len=1，则每次 `all_reduce` 传输 `8192 * 2 bytes (FP16) = 16KB`。TP=4 需要 4 次 `all_reduce`（每层 2 次，共 80 层），总通信量约 `80 * 4 * 16KB = 5MB`。

#### 3.3.2 Pipeline Parallel (PP)

将模型按**层**切分到多个 GPU。

- **Bubble 问题**：PP 中，GPU 空闲等待上游/下游数据传输的时间；
- **1F1B（One Forward One Backward）**：交错调度，减少 bubble；
- **Interleaved 1F1B**：将模型切分成更多 stage，进一步减少 bubble。

### 3.4 批处理策略

#### 3.4.1 Continuous Batching

传统 Batching：一个 batch 内的所有请求**同时开始、同时结束**（受最慢请求拖累）。

Continuous Batching（也叫 Inflight Batching）：
- 新请求可以在**任意时刻**加入 running batch；
- 每个请求独立维护状态，完成的请求随时退出；
- 通过 **batch token budget** 控制总计算量。

**延迟抖动问题**：
- 新请求加入时，需要做 prefill（计算量大），会打断 running 请求的 decode，导致 TTFT 增加；
- **解决方案**：Chunked Prefill。

#### 3.4.2 Chunked Prefill

将长 prefill 拆成多个 chunk，和 decode 交替执行：
- 每个 step 只处理一部分 prefill token + 所有 decode token；
- 避免单个长 prefill 阻塞整个 batch 的 decode。

#### 3.4.3 Disaggregated Serving

2025-2026 年的前沿方向，将 **Prefill 和 Decode 分离**到不同 GPU：
- **Prefill 节点**：高算力，处理长序列的 KV Cache 计算；
- **Decode 节点**：高带宽，处理自回归生成；
- **KV Cache 传输**：通过 RDMA（如 InfiniBand）在节点间传输 KV Cache。

**优势**：Prefill 和 Decode 的资源需求不同，分离后可以各自优化（如 Prefill 用 A100，Decode 用 H100）。

### 3.5 CUDA / Kernel 优化

#### 3.5.1 FlashAttention 原理

标准 Self-Attention 的内存复杂度是 $O(N^2)$，FlashAttention 降到 $O(N)$：

1. **Tiling**：将 Q/K/V 矩阵分块（tile），每次只加载一个 tile 到 SRAM（shared memory）；
2. **Online Softmax**：不需要等所有 $QK^T$ 算完再做 softmax，而是增量计算；
3. **Recomputation**：不存储中间 $S=QK^T$ 矩阵，反向传播时重新计算，用算力换显存。

**性能提升**：标准 Attention 的瓶颈在**HBM 读写**，FlashAttention 通过减少 HBM 访问次数（从 $O(N^2)$ 到 $O(N)$）实现加速。

#### 3.5.2 量化技术

| 量化方法        | 原理                                            | 精度损失 | 适用场景       |
| --------------- | ----------------------------------------------- | -------- | -------------- |
| **FP8**         | NVIDIA Hopper 原生支持，1 byte                  | <1%      | H100/H200 首选 |
| **INT8 (PTQ)**  | 后训练量化，per-channel/per-token               | 1-3%     | 通用部署       |
| **SmoothQuant** | 将激活的 outliers 迁移到权重，减少量化误差      | <1%      | 推荐搭配 INT8  |
| **AWQ**         | 保护 1% 的重要权重（salient weights），其余量化 | <1%      | 4-bit 部署     |
| **GPTQ**        | 逐层量化，用 Hessian 矩阵校准                   | 2-5%     | 极致压缩       |

**面试必问**：AWQ 和 GPTQ 的区别？  
A：AWQ 是**保护重要权重**（基于激活幅度判断），用 per-channel 缩放；GPTQ 是**逐层优化**，用二阶信息（Hessian）校准，精度通常更高但量化速度慢。

#### 3.5.3 CUDA Graph

**问题**：每次 kernel launch 都有 CPU 开销（参数传递、调度），小 batch 时占比高。

**CUDA Graph**：捕获一次完整的 forward 计算图，后续直接**重放**（replay），跳过 CPU 调度。

**限制**：
- 输入输出 shape 必须固定（所以 vLLM 的 CUDA Graph 需要预分配多种 shape）；
- 不支持动态控制流（如 `if` 分支）。

## 第四部分：通用基础 —— Python 并发硬核

不管投哪个方向，这些问题秋招必问：

| 问题                                                  | 工业级答案                                                   |
| ----------------------------------------------------- | ------------------------------------------------------------ |
| `asyncio` 的事件循环怎么实现？                        | 基于 `selector`（epoll/kqueue/IOCP）的 IO 多路复用 + 就绪队列调度。`BaseEventLoop._run_once()` 每次迭代：执行就绪回调 → 轮询 IO 事件 → 处理定时器。 |
| `asyncio.gather` vs `asyncio.wait`？                  | `gather` 等所有任务完成并返回结果列表，其中一个异常默认取消其他任务；`wait` 更灵活，支持 `return_when=FIRST_COMPLETED`，但不会自动取消其他任务。 |
| `asyncio.Semaphore` 公平性？                          | 默认**非公平**。高并发下可能饿死先到请求。生产环境需要自研 `FairSemaphore`（基于 `asyncio.Queue` 维护等待队列）。 |
| 异步中调用同步阻塞代码？                              | `loop.run_in_executor()` 或 `asyncio.to_thread()`。但会占用线程池，需要限制 `max_workers`（通常设为 CPU 核心数）。 |
| `aiohttp` 连接池配置？                                | `TCPConnector(limit=100, limit_per_host=30, ttl_dns_cache=300, use_dns_cache=True, enable_cleanup_closed=True)` |
| 怎么实现背压（Backpressure）？                        | 服务端：`asyncio.Queue(maxsize=N)`，满时 `put` 阻塞或丢弃低优先级消息；客户端：`asyncio.Semaphore` 或 Token Bucket 限流。 |
| 怎么检测客户端断开？                                  | SSE：`await request.is_disconnected()` 或捕获 `CancelledError`。断开后必须调用 `abort_request` 释放 GPU 资源。 |
| 大量 Task 内存泄漏？                                  | Task 异常如果没有被处理，Python 会保留 traceback。务必 `add_done_callback(lambda t: t.exception())`。 |
| `asyncio.create_task()` 和 `await coroutine()` 区别？ | `create_task` 立即提交到事件循环，后台执行；`await` 等待当前协程完成。如果需要并行，必须用 `create_task` + `gather`。 |
