## 一、心智模型：事件循环

### 1.1 为什么要异步？（用自己的话写）

- 同步 IO 的问题是什么？用 Agent 场景举例：__同步 IO 在等待时会一直阻塞整个线程导致 CPU 资源的大量浪费；尤其是在 Agent 场景下，我们的性能瓶颈绝大多数时候都不在计算上，而是等待网络 IO 的响应，也就是调用API的结果，这种场景天然与异步 IO 相性十分好。__
- 三种并发方案的对比（多线程 / 回调 / 协程），各有什么优劣：__三种并发方案分别是 Java、 JavaScript 和 python/Go 选择的方案。Java 使用的是多线程，在底层调用 C 代码进行多个线程的创建和在其中进行切换，优势是显而易见的：这种方案可以使用多核处理器的全部核心，但是劣势也在于此，切换线程是一个特权操作，在操作系统中我们的用户态程序是没有权限进行这样的操作的，我们需要从用户态切换到内核态之后才能切换线程，之后还需要从内核态返回用户态继续执行我们的代码，这样的切换带来的开销是微秒级别的，而且每一个线程都有自己的独立栈空间，也要占用 1MB 的加内存，导致开辟大量线程对于服务器的压力是巨大的，最后由于这样的一个多线程是抢占式执行的，我们无法预测操作系统会如何调度我们的各个线程，所以为了保证数据的一致性，我们需要复杂的竞态分析和锁的使用，在开发中也增加了工程复杂度；JavaScript 使用的回调方案就更加轻量化，不需要进行用户态、内核态的切换，也没有独立栈空间占用服务器宝贵的内存资源，看上去很美好，但是 JavaScript 的工程复杂度也是极大的，因为随着回调层数的增加对于开发者心智的占用呈指数型增长，同时难以处理庞杂回调结构中的错误信息；python 使用的是一种称为协程的方案，也在 Go 和其它语言中使用，python 中的协程是一种比回调更加轻量的方案，即不需要用户态和内核态的切换，也没有栈空间占用内存空间，而且作为一种协作式的并发方案，开发者可以确定并发时的执行顺序，不但便于开发和调试，同时错误的传递也较为简单，代价就是 async 的传染——一旦进入了异步空间，一切都只由事件循环主导，而事件循环是不知道同步代码的存在的。__
- Python 为什么选了协程这条路？__python 的协程最早是从 python 2 时期的生成器而来的，生成器时期我们可以通过 yield 关键字来返回值并保存当前函数的断点和栈帧，之后使用 next() 可以从断点处继续执行；在 python 3 的早期，随着父子生成器的引入，yield from 关键字可以将控制权转交给后面的生成器，大体的协程模型已经完成了，但是生成器有自己的问题—— yield 关键字的语义混乱，这个关键字不但可以将值传递到外部，类似 return ，还可以从外部获取值进入函数体，还可以向外部抛出异常，所以 asyncio 用 await 关键字等方法代替了 yield，使得语义统一。__

### 1.2 事件循环是什么？（用自己的话写）

- 用一句话描述事件循环：__事件循环是异步 IO 中唯一的控制者，进入异步 IO 一切调度控制全部由事件循环中的定义决定__。
- 事件循环内部的两个核心数据结构是什么？它们分别存什么？__就绪列表是只要控制权在事件循环就开始顺序执行的列表，等待字典是一个等待条件：回调方法组成的表，在 check_io 中检测到满足了条件后就移除这个条目，添加到就绪列表。__
- 事件循环一轮循环（一个 tick）干了哪些事？__先执行就绪列表中第一个 task 执行一步，然后检查等待字典有没有 IO 事件就绪，把就绪的放到就绪列表，返回第一步__
- 为什么事件循环是单线程的？单线程意味着什么？__因为 python 的异步 IO 是协作式的，需要当前协程主动让出才能切换，所以事件循环使用的是单线程，单线程意味着执行顺序是可预测的。__

### 1.3 并发（Concurrency）vs 并行（Parallelism）

- 用你自己的例子说明两者的区别：__并发是在一段时间内不同的任务交错进行；并行是一段时间内不同的任务同时进行，硬性需要多个逻辑处理单元。__
- Agent 开发中为什么 99% 场景用并发就够了？__因为 Agent 场景我们最大的等待时间不是计算而是 IO 等待，所以我们只要利用等待的时间进行其它需要 CPU 资源的操作就完全可以完成；Agent 不是一个计算密集型的场景。__

## 二、async / await 语法

### 2.1 `async def` 创建了什么？

- 调用 `async def` 函数会发生什么？返回的是结果吗？__直接调用 async def 函数不会执行其中的代码，而是会返回一个对应的协程对象。__
- 协程对象包含哪些东西？__里面有整个函数体的字节码和干净的栈帧。__
- 为什么说 `async def` 是"协程工厂"而不是协程本身？__因为调用 async def 函数生成的是协程对象而不是执行协程中的代码__

### 2.2 `await` 做了什么？

- `await` 是不是"阻塞等待"？如果不是，它是什么？__await 做的是交出 CPU 控制权，把控制权交给事件循环或者是协程对象__
- 描述 `await` 时控制权是如何一层层传到事件循环的：__执行 await 时，当前协程挂起，一路 yield None 当 run 收到 None 的时候就知道整个链路已经被挂起了，将控制权交给事件循环去执行其他协程。__
- `await` 后面能跟什么？列出四类 awaitable 对象：
  
  1. ________________
  
     协程对象，直接执行协程对象中的代码
  2. ________________
  
     task 对象（future 子类），~~直接放入就绪列表，等待下一次控制权交给事件循环~~<span style="color:red">包装了协程对象的 Future 子类，由 create_task 自动注册到事件循环，await 时等待内部协程跑完并取返回值</span>
  3. ________________
  
     future 对象，本身是一个空的容器，等待其他协程往其中写入 result
  4. ________________
  
     所有实现了 .__await\_\_() 函数的对象，但不常见

### 2.3 协程的状态机

- 协程有哪四种状态？各代表什么含义？
  1. ________________
  
     CORO_CREATED  协程已经被创建
  2. ________________
  
     CORO_RUNNING 协程正在执行
  
  3. ________________
  
     CORO_SUSPENDED 协程已经被 await 挂起
  
  4. ________________
  
     CORO_CLOSED 协程已经执行完毕，返回或者抛异常了
- 状态标记的作用是什么？（提示：防止什么非法操作？）__防止同一个协程对象被多个地方同时驱动导致栈帧混乱__

### 2.4 Future 和 Task

- Future 是什么？用一个生活中的类比解释：__future 是一个空的信封，等待其他协程给它放入信纸。__
- Task 和 Future 是什么关系？Task 多了什么能力？__Task 是 Future 的子类，Task 具有自己跑完协程之后把返回值填入 result 的能力。__
- `await task` 的本质是什么过程？__本质是等待这个 Future 被填上值，然后取出那个值。 Task 可能已经在前面的控制权转移后台跑了，await 只是停下来等结果。__
- Future / Task / Coroutine 三者的层级关系图（用箭头画）：__Future -(派生类)> Task <(包装成)- Coroutine__

### 2.5 `asyncio.run()` 干了什么？

- 它的四步流程是什么？
  1. ________________
  
     创建一个事件循环，并且设为当前线程的事件循环
  2. ________________
  
     把参数中的协程对象包装成 Task ，放到事件循环的就绪列表中
  3. ________________
  
     运行所有 pending 的 callback，等待所有的 finally 块执行完毕
  4. ________________
  
     关闭事件循环
- 为什么不能在已有事件循环的地方再调 `asyncio.run()`？__因为每一个线程只能有一个当前事件循环，再次调用会导致报错。__

### 2.6 🚨 易错：`asyncio.sleep()` vs `time.sleep()`

- 两者的本质区别：__asyncio.sleep() 是挂起当前协程由事件循环继续执行其他协程；time.sleep() 是阻塞整个线程等待，期间没有任何任务执行。__
- 如果在 async 函数里用了 `time.sleep(5)`，会发生什么？__会导致整个线程阻塞 5s，所有协程全部停止运行__
- 为什么？__因为不会转交控制权到事件循环，就阻塞在当前协程，事件循环也不能去调度其他就绪协程执行。__

### 2.7 async 的传染性

- 什么是"async 传染"？__当调用一个 async def 的时候，这个def 也需要被 async 修饰 。__
- 同步世界进入异步世界的唯一入口是什么？__asyncio.run()__

### 2.8 与 Java 多线程的对比

| 维度 | Java Thread | Python asyncio |
|------|------------|----------------|
| 并发单元 | 线程 | 协程 |
| 调度者 | 操作系统 | 事件循环 |
| 切换时机 | 时间片结束时 | 当前协程主动转交控制权 |
| 切换开销 | 用户态到内核态到用户态 ~ 40μs | 极小 ~ 100ns |
| 数据共享 | 线程间栈不共享 | 栈空间共享 |
| CPU 密集型 | 多核同时使用 | 本身是单线程，但是可以开辟一个新的线程专门处理重计算任务 |
| IO 密集型 | 开辟大量线程会占用大量内存，频繁切换线程会有极大开销 | 单线程高效处理，大量协程占用极少内存 |

## 三、并发工具：gather / create_task / wait

### 3.1 三件套速查表

| | gather | create_task | wait |
|---|---|---|---|
| 启动时机 | 调用 gather 时，await 将控制权转交给事件循环 | 等到下一次当前协程将控制权转交给事件循环 | 调用 wait 之后将控制权转交给事件循环 |
| 结果顺序 | 按照传入的顺序 | 按照执行结束的顺序 | 无序 |
| 异常行为 | return_exceptions=<br />False：第一个异常~~全部取消~~<span style="color:red">立即传播，其余任务继续执行但结果被丢弃（不是取消！）</span><br />True：报错当作正常返回值存在列表 | 每一个单独处理 | 全部保存在返回的两个 Task 对象列表中，需要自行取出处理 |
| 取消行为 | 取消 gather 自动将所有参数全部取消 | 每个单独处理 | 不做任何取消，必须手动取消 |
| 适用场景 | 需要要么全部成功，要么全部取消的场景 | 需要每一个单独处理的场景 | 需要进行精细化操作的场景 |
| 返回值 | result 列表 | Task 对象 | 一个已经完成了的 Task 对象列表，一个没有完成的对象列表 |

### 3.2 `gather` 深入

- `gather` 对传入的协程做了什么？协程和 Task 的处理一样吗？__不一样，对于协程对象， gather 会将其封装成为 Task 对象再添加到就绪队列；对于 Task ~~直接添加到事件循环的就绪队列~~<span style="color:red">不再包装，直接等待它完成（Task 本身早已在事件循环中）</span>__
- 🚨 易错：gather 默认异常行为是什么样的？成功的结果能拿到吗？__默认的异常行为是有一个报错~~，全部取消~~<span style="color:red">立即传播异常，其余任务继续跑但结果被丢弃</span>，成功的结果也拿不到__
- `return_exceptions=True` 改变了什么？__改变了对于异常行为的处理方式，现在遇到异常行为静默处理，把错误对象当作正常的返回值处理__
- gather 的取消是如何传播到子任务的？__首先对 gather 自身标记取消，之后对于传入参数的对象进行遍历，每个都执行取消操作，最后取消自己__

### 3.3 `create_task` 深入

- Task 是什么时候开始执行的？是在 `create_task` 时，还是在 `await` 时？__~~是从 await 时候才开始执行的，但是不一定需要 await 这个 task，await 任意都可以，主要是需要这个控制权转交的行为~~<span style="color:red">在 create_task 时被调度（放入就绪队列），在当前协程 await 交出控制权后开始真正执行——不一定需要 await 这个 Task 本身，await 任何东西都能交出控制权</span>__
- 🚨 易错：创建了 Task 但不保存引用，可能导致什么问题？__可能会导致出现异常但是没有进行 finally 代码块的释放就被事件循环强制关闭__
- `TaskGroup`（3.11+）解决了什么问题？和 gather 有什么不同？__解决了必须要保存 Task 引用的问题，使用 TaskGroup 会自动管理资源的释放，并且任何一个 Task 报错的时候都会自动取消全部 Task__

### 3.4 `wait` 深入

- `return_when` 的三个值分别是什么行为？
  
  1. ________________
  
     ALL_COMPLETED: 所有 Task 都完成或者是超时才返回
  2. ________________
  
     FIRST_COMPLETED: 只要有一个 Task 完成就返回
  3. ________________
  
     FIRST_EXCEPTION: 只要有一个 Task 抛出异常就返回
- `wait` 的返回值是什么结构？和 `gather` 的返回值有什么本质不同？__wait 返回的是两个 Task 对象列表，想要获取到对应的返回值需要自己调用 result__
- 🚨 易错：`wait` 的异常处理方式和 `gather` 有什么关键区别？__wait 本身不对异常做任何处理，需要自行从返回列表中取出有异常的 Task 再取出异常才能进行处理__

### 3.5 实战决策树（用自己的话重画）

```
你有一组协程要并发跑
│
├─ 需要所有结果，顺序保持，一个失败全失败 → 用 gather
├─ ~~需要所有结果 → 用 wait(return_when=ALL_COMPLETED)~~<span style="color:red">需要所有结果，个别失败不影响 → 用 gather(return_exceptions=True)</span>

```

### 3.6 并发限流：Semaphore

- 为什么要限流？（Agent 开发中的具体场景）__同时对 API 发送过多请求会导致被限流，或者批量访问大量 URL 导致被 ban 或者给服务器打崩了__
- Semaphore 的内部原理（一句话）：__内部维护一个计数器，当有请求进行时就会占用一个计数，如果计数器为 0 就阻塞其他的任务继续获取许可，有请求完成释放计数就在计数器 +1 。__
- `async with sem` 做了什么？__对同时发起的请求计数，将多余的请求先阻塞不要发出__

## 四、httpx 实战

### 4.1 为什么选 httpx？

- httpx vs requests vs aiohttp 的对比：__httpx 原生支持 asyncio 并且支持 HTTP2 协议__
- Agent 开发中选 httpx 的三个理由：__支持 asyncio，支持 HTTP2，更好的超时制和连接管理__

### 4.2 同步 Client 是什么？

- `httpx.Client` 内部维护了哪些东西？__维护了一个连接池，以及默认的 user-agent，还有超时配置和 cookie 管理，重定向等__
- 为什么要用 `with`？不用会发生什么？__用 with 是为了自动管理资源的创建和释放，不用可能会导致资源的泄露，本地的文件描述符被耗尽__
- 单次 HTTP 请求的完整生命周期（从 DNS 到返回响应）：__DNS 解析，TCP SSL 握手，发送请求体，接受响应__
- 连接复用为什么重要？__因为一个 http 的连接需要 TCP SSL 握手，一次连接的建立就在 100ms 左右，复用连接可以显著缩小这部分开销__
- httpx.Timeout 的四个维度分别是什么？
  1. ________________
  
     connect：建立 TCP SSL 连接的超时时长
  2. ________________
  
     read：接收到服务端发送第一个字节的超时时长
  3. ________________
  
     write：发送请求体的超时时长
  4. ________________
  
     pool：从连接池获取连接的超时时长

### 4.3 异步 AsyncClient

- `Client` 和 `AsyncClient` 的 API 对应关系（三个关键词的不同）：__使用 async with 代替 with，await client.get() 代替 client.get()__

- `client.get(url)` 返回的是什么？这一行有没有真正发请求？__返回是一个协程对象，在这一行没有发送请求__
- 🚨 易错：为什么不能在 async 函数里用同步的 `httpx.Client`？__两者底层实现完全不同，同步的 Client 不走事件循环__

### 4.6 异常处理和重试

- httpx 的异常层次结构（画出来）：

  httpx.TimeoutException  (所有超时的基类)

    └── httpx.PoolTimeout      (等待连接池的空闲连接超时)

- LLM 调用场景最常见哪种超时？该设多大？__读超时设置大一些， 60-120s 左右__

- 指数退避重试的逻辑（用自己的话写伪代码）：

  ```python
  for attempt in range(retry):
      try:
          r = await client.get(url)
          r.raise_for_status()
          return r
     	except (httpx.TimeoutException, httpx.ConnectError) as e:
  ```

- 4xx 和 5xx 错误的重试策略有什么不同？为什么？__因为 4xx 的错误码是客户端错误，表示用户请求有问题，重试也是有问题； 5xx 是服务端问题。__

### 4.7 流式响应

- 什么是流式响应？Agent 调用 LLM 时为什么需要它？__流式响应是与普通的响应相对的，普通的响应等待服务端完全准备好发送响应；流式响应在 Agent 场景下是 LLM 每推理出一个 token 就发送给客户端，可以避免长时间等待，可以看到逐 token 的响应__
- 同步版和异步版的迭代方法分别叫什么？__同步迭代是 iter_lines，异步迭代是 aiter_lines__
- `async for` 和普通 `for` 的区别？__async for  每次有数据传入就 yield，没有数据传入的时候事件循环继续处理其他协程，不阻塞；for 中间不能调度其他协程__

## 六、常见陷阱汇总

- [x] `time.sleep()` 代替 `await asyncio.sleep()` — 阻塞整个事件循环
- [x] 创建 Task 不保存引用 — 异常被静默吞掉
- [x] `gather` 一个抛异常，其余结果全丢 — 需要 `return_exceptions=True`
- [x] `wait` 和 `gather` 异常行为不同 — `wait` 不传播异常
- [x] 并发不设上限 — API rate limit / 文件描述符耗尽
- [x] 在 async 函数中混用同步 `httpx.Client`
- [x] 忘记 `await` — 协程没被驱动，什么都没发生
- [x] 在已有事件循环的地方调 `asyncio.run()` — 报错
- [x] 异步生成器未关闭 — `asyncio.run()` 结束时自动清理，但手动管理时要小心

## 七、一句话总结（考试级精简）

- 事件循环：__异步 IO 中唯一的调度者，维护就绪列表和等待字典__
- async def：__协程对象工厂，调用产生协程对象__
- await：__CPU 控制权转交__
- Task：__自动加载到就绪列表等待下一次控制权在事件循环身上运行__
- Future：__空的容器，等待其他协程往 result 中写入值__
- gather：__批量包装协程对象到事件循环中__
- create_task：__将协程对象包装成 Task 并注册到事件循环__
- Semaphore：__用于限制并发数量的信号量__
- httpx.AsyncClient：__异步 http 客户端，维护连接池，超时配置，cookie 管理，重定向相关内容__
