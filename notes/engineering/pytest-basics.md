## 1. fixture — 依赖注入

<!-- 用自己的话写：fixture 是什么，pytest 怎么找到它 -->

- fixture 类似于 Java spring 中的 Bean 类，进行自动的依赖注入，是我们定义的一个可以服用的样板，让我们在后续的测试中不需要反复地创建对象，如果大量重复创建对象 ，在这个类修改之后我们需要对每一个构造函数修改，但是用了 fixture 之后我们只需要修改一次就可以了，其他调用 fixture 的地方不需要处理；pytest 通过我们在函数上加 @pytest.fixture 装饰器的方式来找到这样的 fixture

**scope 控制生命周期**：

| scope | 含义 |
|-------|------|
| function | - 每次调用时创建一个实例 |
| class | - 对每个测试类使用同一个实例，没用到 class TestXxx: 就没作用 |
| module | - 每一个模块中创建一个实例，模块内复用 |
| session | - 每次运行时创建一个示例 |

**teardown（yield）**：

<!-- 用 yield 怎么做到测试后自动清理 -->

- 我们在 yield 之前进行预处理等操作，yield 之后相当于将需要的实例返回了，在最后退出的时候，回到 yield 执行之后的清理操作

## 2. parametrize — 参数化测试

<!-- 用自己的话写：parametrize 解决什么问题 -->

- 解决我们想要对同一个地方测试多组数据的问题，不使用 parametrize 我们要测多少种就需要写多少个测试函数，用了之后我们只需要将所有需要测试的参数列表写入就可以通过一个函数测试整个参数列表

**多个 parametrize 叠加**：

<!-- 效果是什么 -->

- 多个 parametrize 叠加的效果是对这些参数进行笛卡尔积运算，也就是相当于多个遍历，找出每一种组合全部测试一遍

## 3. pytest.raises — 异常断言

<!-- 用自己的话写：pytest.raises 的用法和注意事项 -->

- pytest.raise 的作用是认为报错是正常行为，正常运行认为是错误，也就是我们认为这个操作正常就是要报错的时候使用；使用时要注意我们 with pytest.raises(SomeError): 里面是可能抛出错误的代码，如果可能抛出错误的代码在此之前，那么运行到这里时，要么已经报错退出，要么不会再报错，raises 都检测不到。

## 4. mock — 替换外部依赖

<!-- 用自己的话写：mock 是什么，什么时候用 -->

- mock 是一个替换真实业务中的内容使用的一个虚假容器，这个容器内可以关联任何内容并替代，当我们在测试时出于时间、成本等因素的考量，想要快速完成测试的时候使用。

**patch 路径规则**：

<!-- 路径写错是最常见的坑，用自己的话总结 -->

- patch 路径是调用方所使用的路径，比如说我们有一个 add 函数，定义在 math.py 中，在 compute.py 中 from math import add，那么我们在测试函数中进行 mock 的时候  patch("src.compute.add", return_value=something)，大概就是这样的；如果在被调用方处 mock ，就相当于改的是工厂里的流水线，但是用户拿到的已经是原来流水线生产完的产品，自然是无法生效的。

**stub vs mock**：

| | 是什么 | 做什么 | 验证什么 |
|---|------|--------|---------|
| Stub | - 一个虚拟空容器 | - 替换真实值 | - 什么都不验证 |
| Mock | - 一个虚拟的空容器，可以放任意内容 | - 代替真实值 | - 可以验证是否被调用，被调用几次 |

## 5. 常见错误

<!-- 今天踩过的坑，记下来以后别再踩 -->

- pytest 检测的测试文件在 tests 目录下，文件名为 \*\_test.py 或 test_*.py
- pytest 检测的测试函数名称是 test_*
- 导入一个文件内的多个类就直接一个个写，这样能快速看出用了哪些内容

## 6. 常用命令速查

```bash
uv run pytest -v					# 运行全部测试
uv run pytest tests/xx.py -v		# 测试单个文件
uv run pytest tests/xx.py::def_name	# 测试单个函数
uv run pytest --collect-only 		# 只收集不运行
```
